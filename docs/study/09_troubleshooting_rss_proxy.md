# 🛠️ 실전 디버깅 전략: 500 에러와 RSS 우회 전술

개발 중 발생하는 500 내 서버 오류(Internal Server Error)는 기술적으로 가장 막막한 순간 중 하나입니다. 이번 멜로디쉐어 프로젝트에서 인기 차트 기능을 구현하며 우리가 해결한 과정을 통해 '문제 해결 노하우'를 학습합니다.

## 1. 문제의 발단: 500 에러와 `cheerio`
- **상황**: `yt-search` 라이브러리를 사용해 차트를 불러오려 했으나 500 에러 발생.
- **원인 분석**: 브라우저 콘솔 및 서버 로그를 통해 `Error: Cannot find module 'cheerio'` 문구 확인.
- **결론**: 라이브러리 내부에서 사용하는 특정 모듈이 Next.js 앱 라우터(App Router)의 서버 환경(Runtime)과 충돌하여 로드되지 않는 기술적 결함이 있었음.

## 2. 해결을 위한 사고 과정 (Thinking Process)
문제가 발생했을 때 개발자는 다음과 같은 단계를 거쳐야 합니다:
1. **재설치 및 재환경 설정**: `npm install`이나 `next.config` 수정을 시도함 (우리가 시도했던 것).
2. **한계 인정**: 환경 설정으로 해결되지 않는다면 '도구 자체'의 호환성 문제일 가능성이 높음.
3. **대안 탐색**: "차트 데이터를 얻는 목적은 같지만, 라이브러리를 쓰지 않는 다른 방법은 없을까?"
4. **우회로 발견**: 유튜브는 플레이리스트를 'RSS 피드'라는 정형화된 XML 데이터로 제공한다는 점에 착안.

## 3. RSS 피드를 활용한 'Zero-Dependency' 전략
- **안정성**: 외부 라이브러리(`yt-search`)에 의존하지 않고, 브라우저 표준 기능인 `fetch`만으로 데이터를 가져옴.
- **성능**: 복잡한 스크레이핑(Scraping) 과정이 없어 속도가 매우 빠르고 서버 부하가 낮음.
- **유지보수**: 라이브러리 업데이트에 따라 코드가 깨질 걱정이 없음.

## 💡 당신의 실력을 높여줄 노하우
> [!TIP]
> **"돋보기가 고장 났을 때, 돋보기를 고치려 하지 말고 안경을 써보세요."**  
> 특정 라이브러리가 안 될 때, 그 라이브러리를 고치는 데 너무 많은 시간을 쓰기보다 **'데이터의 원천(Source)'**을 직접 다루는 방법(RSS, Raw API 등)을 고민하는 것이 시니어 개발자의 문제 해결 방식입니다.

## 📚 더 공부하면 좋은 키워드
- **RSS (Really Simple Syndication)**: 콘텐츠 업데이트 정보를 전송하는 XML 기반 포맷.
- **Regex (정규 표현식)**: 복잡한 문자열(XML/HTML)에서 필요한 데이터만 쏙쏙 뽑아내는 강력한 기술.
- **Next.js Route Segment Config**: `export const runtime = 'nodejs'` 등 서버 환경을 제어하는 설정들.
